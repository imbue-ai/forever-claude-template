# Live editing flow for the system interface

Rework the system-interface editing flow so it mirrors `update-service`'s "live loop first, ratify at turn-end" pattern, collapsing the difference onto three system-interface-specific adjustments. Also clarify the mock guidance shared by all interactive deliveries along the way.

## Overview

- Today `update-system-interface` sends every change to a background worker that must pass the **full** build + test + review gate **before** the user's first preview — so iterating on the workspace UI means waiting out a whole hardening pass per round. This inverts the repo's own "live first, ratify at turn-end" principle.
- The fix: keep `update-system-interface` as the front door — skills route on description text, and the workspace-UI triggers (dockview, chat interface, progress view) route best from a dedicated description — but gut its divergent flow and rewrite it as the system-interface specialization of `update-service`'s live-loop-then-harden shape. The system interface becomes "the service that *is* the workspace UI"; the skill body carries only what differs, and everything genuinely shared (lease mechanics, the mock taxonomy, the turn-end harden handoff) lives in `update-service` and the shared references it points at.
- The difference collapses onto **three adjustments**, each traceable to "a broken build is served straight to the user as their whole workspace": (1) **code isolation** — edit an isolated worktree, never the served tree; (2) the **isolated preview instance is the always-on user's view** (not an escalation, as it is for a normal service); (3) **safe-reveal** (health-checked, auto-rollback) as go-live.
- The live loop is **lead-driven and cheap**: edit worktree → build → refresh the preview in place → iterate with the user; the expensive test/review gate is deferred out of the loop and only runs once, after the user approves.
- Mock guidance is generalized for all interactive deliveries: two distinct demonstrative-artifact types (janky real edit vs. detached prototype), with when-to-use guidance keyed on wiring-cost vs. restart-cost — clarifying that the mock's purpose is **fast feedback**, not just breakage-avoidance. The taxonomy lands in `interactive-delivery.md` (it refines that skeleton's phase 5); `update-service` applies it with service-flavored examples.

## Expected behavior

- **Iterating on the workspace UI is fast.** The user sees and reacts to a change after a build (seconds), not after a full harden pass (minutes). The heavy gate runs once, after they're happy.
- **The one real up-front cost hides behind exploration.** On entering the flow the lead immediately starts worktree provisioning as a background task — `git worktree add` on the pass's branch, dependency sync (`uv sync --all-packages`, `npm ci`), first build — and spends that time reading code and pinning down the shape with the user. By the time there is an edit to show, the worktree is warm.
- **A change is delivered through one lead-driven worktree+preview loop:** edit → build → refresh the isolated preview → surface. The lead commits before each surface, so branch HEAD always equals the last thing the user saw.
- **The lead owns the live-phase worktree outright; the worker owns its own.** The lead's editing space is a plain `git worktree add` on the pass branch (`mngr/update-$SLUG`) — no worker exists during the live phase, so there is no shared-worktree ownership contract to maintain. The harden worker is created only at approval, in its own fresh worktree checked out on that same branch.
- **The preview refreshes in place; the user's tab never goes blank.** The tab points at the wrapper page, which never moves. A frontend-only round is rebuild + `layout.py refresh si-preview` (the existing iframe-reload command). A backend round additionally bounces the inner app process on the same port via a new `preview-refresh` sub-command — reusing `preview`'s boot and health-check code — leaving the wrapper and the tab untouched.
- **The isolated preview is always the user's view during iteration.** Unlike a normal service (where opening a second tab beside an already-open one is awkward), a labeled preview tab reads naturally for the whole-workspace UI. The wrapper/labeling machinery is unchanged from today's flow; only the in-place refresh is added.
- **Every system-interface change runs through the isolated worktree** (code isolation is always required). Only the preview/mock behavior varies:
  - Visual changes: iterate live in the preview.
  - Backend logic: the preview boots to let the user verify behavior.
  - Test-only / no-surface changes (e.g. a test-suite fix): skip the preview entirely → edit worktree → harden → merge → safe-reveal.
- **How rough the first previewed pass is scales with shape-uncertainty**, not "does it change what the user sees." Obvious contained changes (font, color, reposition, copy) are implemented directly; redesigns / new views / non-obvious layouts start with a deliberately rough pass for fast signal.
- **Two demonstrative-artifact types are available** (for the system interface and for ordinary services):
  - *Type 1 — janky real edit:* rough/hardcoded but in the real code (worktree), shown via the real preview. Faithful; flows straight into the real implementation.
  - *Type 2 — detached prototype:* a separate fake UI. Cheapest/fastest, no real wiring, thrown away; may "not convince" when real rendering matters.
  - Choose by **wiring-cost vs. restart-cost**: Type 2 for fastest look-and-feel / comparing directions when a fake conveys it and real wiring is costly; Type 1 when real context is needed to convince, integration is cheap, or the rough version should flow into the real thing. The embedded system-interface UI defaults to Type 1; Type 2 is reserved for genuinely standalone new surfaces.
- **Handoff to the worker depends on whether the approved artifact is real code yet:**
  - *Type 1 (janky real edit):* hands off directly — the worker's task is "implement this approved shape for real, then harden."
  - *Type 2 (prototype):* the lead first builds it for real in the worktree and does a live-preview round (real context), then hands off to "harden."
- **A final preview-before-merge is kept when the worker produced real work the user hasn't seen** (Type 1 janky → real: worker done → lead previews the worker's output → merge + safe-reveal) **and the change warrants a preview at all** — the system interface always warrants it; other services only sometimes (a tab refresh after go-live often suffices). It is optional when the user already previewed a polished real version.
- **Only one system-interface edit happens at a time:** the editing lease is held from flow entry through reveal or abandonment (a deliberate divergence from `update-service`'s per-turn lease release). The stale-holder break path is the same as `update-service`'s — the lease is advisory; a holder whose agent is dead or long-silent is surfaced to the user and broken only by their call — with the addition that breaking it also means tearing down the orphaned pass: the preview service and its tab, the lead's worktree, and the worker if one exists. The worktree/preview persist across turns; if the user never approves, explicit abandonment tears everything down (no idle timeout).
- **Go-live is unchanged:** the health-checked, auto-rollback safe-reveal.

## Changes

- **Keep `update-system-interface` as the front door, rewritten thin.** Its description (with the workspace-UI routing triggers) stays; its body becomes the live-loop specialization — the three adjustments plus the SI-specific mechanics — pointing at `update-service` and shared references for everything shared. The old delegate-then-preview flow is removed.
- **`reveal_system_interface.py` stays in place**, and `preview` / `unpreview` / `reveal` stay unchanged. Add one `preview-refresh` sub-command: restart the inner app process on the same port (for backend rounds) and re-run the existing health check, never touching the wrapper page or the tab. Frontend-only rounds need no script change at all (rebuild + the existing `layout.py refresh si-preview`).
- **Lead-driven live loop replaces "worker hardens before first preview."** The lead edits the isolated worktree, builds, refreshes the preview, and iterates with the user; the full test + review gate is deferred until the user approves.
- **Provision the lead's worktree up front, in the background.** On flow entry, run `git worktree add` (branch `mngr/update-$SLUG`) plus dependency sync and first build as a background task while the lead explores the code and clarifies the change shape with the user. No worker is created during the live phase.
- **Create the harden worker at approval time, on the existing branch.** Add a branch-spec passthrough to `launch-task`'s `create_worker.py` (mngr's `--branch BASE` without `:NEW` checks out an existing branch directly) so the worker's fresh worktree checks out `mngr/update-$SLUG` instead of branching anew from the lead's HEAD. The worker re-syncs its own worktree after approval, in the background, where nobody is waiting. This replaces the previously considered "create the worker up front without sending its task" option — `launch-task`'s task delivery is unchanged.
- **Define the two worker-handoff shapes:** "implement the approved shape for real + harden" (Type 1 janky), and "harden only" / committed-origin verify (an already-polished or already-real-and-previewed change).
- **Keep the final preview-before-merge conditionally** — gated on both "the worker produced unseen real work" and "this change warrants a preview" (always for the system interface; sometimes for other services).
- **Generalize the mock guidance in `interactive-delivery.md`, not `update-service`:** the two demonstrative-artifact types and the wiring-cost-vs-restart-cost choice refine the skeleton's phase 5, so they land there, with the wording tightened so the mock's fast-feedback purpose is explicit (not only its fidelity-escalation). Amend phase 5's "no commits yet" to "no commits on the served branch, no production state" — commits on an isolated throwaway branch are explicitly fine (the live loop depends on them). `update-service` applies the taxonomy with service-flavored examples and documents "does this change warrant a final preview at all?" as a general judgment.
- **Hold the editing lease across the whole pass** (flow entry through reveal or abandonment) for the system interface, diverging from `update-service`'s per-turn release; document the stale-holder break path (same advisory mechanics as `update-service`, plus teardown of the orphaned preview/tab/worktree/worker); persist worktree/preview across turns; explicit-abandonment teardown only.
- **Update cross-references to match the new split:** `update-service`'s "for the workspace UI use update-system-interface" pointer now describes that skill as this flow's system-interface specialization (not a separate delegate-first flow); CLAUDE.md's routing keeps pointing at `update-system-interface`; any other mention of the old delegate-then-preview behavior is reworded.
