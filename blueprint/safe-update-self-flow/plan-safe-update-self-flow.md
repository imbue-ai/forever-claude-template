# Plan: Safe, background-worker-driven `update-self` flow

## Refined prompt

> **Overhaul the `update-self` skill into a safe, background-driven update flow.**
>
> * Today `update-self` does a blind `git pull --no-ff upstream main` in the lead's served tree, with no service validation and no reveal/restart — so new code (e.g. a `.mngr/settings.toml` change the running `system_interface` doesn't yet support) is never actually applied and can break the live workspace.
> * **Default target = latest stable `minds-v*` tag** (excluding `-rc`); the user may override to a specific tag or `main`.
> * All update work is **dispatched to a background worker** (its own branch + worktree), so the live versions are never at risk mid-update. A **new dedicated update-self orchestration** owns the flow end-to-end, but **maximally reuses / generalizes** existing machinery (`launch-task`, `serve_isolated_instance.py`, `reveal_system_interface.py`).
> * **Pre-merge:** worker diff-reasons *and* runs a trial merge (`git merge --no-commit --no-ff`) to enumerate the real conflict set, auto-resolves clear-cut cases (keep-local for agent-specific files, take-upstream for untouched-locally files), and surfaces only genuinely unresolvable ambiguities via a mid-flight `question` gate; the lead relays them to the user, and the worker resumes with the user's resolution.
> * Worker preserves the `update-self:`-subject merge commit on its branch so the lead can land it with the marker intact (lead re-creates it only if history got mangled).
> * **Validation is scoped to the *merged* set** (files where local ended up different from upstream), not clean pull-ins: full test suite + lint + ratchets for impacted projects, isolated-service boots, and **Playwright live-testing of any impacted web surface — `system_interface` or a user web service — whenever the merge reconciled real divergence (non-trivial integration), not a clean pull**.
> * Worker **generates the "what's new" report** (driven by new `changelog/` entries + its own conflict / impacted-service notes) and reports back.
> * Lead outlines what's new + confirms the merge resolution with the user as a **hard approval gate**, then reveals by change class:
>   * `system_interface` real-merge → **preview + `reveal_system_interface.py`** (which rebuilds the gitignored `static/` bundle); clean pull → **reveal only, skip preview**.
>   * service / `supervisord.conf` / `bootstrap` → `mngr start --restart system-services` (+ tab refresh), lead's judgment on whether a preview is worth it.
>   * editable-tool (`vendor/mngr`, `system_interface`) → code picked up live; **manifest (`pyproject.toml`) change → env refresh** (`uv sync --all-packages` / `uv tool install -e … --reinstall`).
>   * `Dockerfile` → **split by what changed, not monolithically rebuild-only.** Changes with a live-applicable equivalent are applied in the running container (e.g. a `CLAUDE_CODE_VERSION` bump → re-run the `scripts/setup_system.sh` install to the new version; note it's coupled to `agent_types.claude.version` in `.mngr/settings.toml`). Genuinely image-level changes (base `FROM`, baked `apt-get` system packages, build-time filesystem layout) have no live equivalent → **surfaced to the user for a manual workspace rebuild**. Worker judgment decides which hunks fall in which class.
>   * skills / `CLAUDE.md` / `.agents/shared/` / `scripts/` → **only "nothing to reveal" when nothing live depends on them at runtime.** These paths are frequently runtime dependencies of live services: every `supervisord.conf` program shells out to `scripts/*.py` (`forward_port.py`, `oom_tag_service.py`, …), and a crystallized `fetch-process-show` / `build-web-service` pipeline lands its scripts under `.agents/skills/<skill>/scripts/`, which a scheduled job or daemon may invoke. The worker must **trace downstream consumers** of any changed shared script/skill; if a live service or runner depends on it, treat it as service-impacting (validate + restart the affected service), not a silent merge.
> * Flow is **single-flight** and requires a **clean tree** (surface and stop otherwise). If the worker gets stuck or dies, surface via the standard worker-failure path; nothing is merged or revealed and the live workspace is untouched.

## Overview

- The problem is that `update-self` today merges upstream directly into the live served tree and stops there: no conflict triage beyond "resolve if needed," no validation that services still boot, and — most critically — no application step, so merged code (settings the running `system_interface` can't parse, a bumped `vendor/mngr`, a new service) is never actually made live and can silently break the workspace.
- The fix mirrors the safety model already proven in `update-system-interface`: never mutate the live tree from an unverified state. All merge + validation work happens in an **isolated background worker** (own branch + git worktree via `launch-task`), and only a known-good, user-approved result is landed and revealed.
- The default target changes from `origin/main` to the **latest stable `minds-v*` tag** (released, already-tested code), with an explicit override for a specific tag or `main`.
- Conflict handling is **triage, not blanket surfacing**: the worker auto-resolves the deterministic cases and gates only genuine ambiguities, so the user is interrupted only when a real decision is needed.
- Work is **scoped by what actually merged**, but the merged-vs-pulled distinction governs **validation depth only, not application**: a clean upstream pull-in (no local divergence) is trusted as upstream-tested and gets minimal validation (and only a *real* `system_interface` merge triggers the human preview) — yet a clean pull-in of a **live runtime dependency still gets applied**, i.e. the services that depend on it are restarted. "Trust, skip heavy testing" never means "skip applying it"; the current blind-pull skill's core bug is precisely that it merges and never restarts.
- Reveal is **classified by change type** and reuses the existing safe-reveal script for `system_interface` (pre-flight boot, health probe, `static/` rebuild, auto-rollback); non-live-applicable changes (Dockerfile) are surfaced rather than silently merged.
- The `update-self:` merge-commit subject marker is preserved end-to-end so downstream built-in-vs-user classification (e.g. `assist`) keeps working.

## Expected behavior

- User asks the lead to update (or names a version). The lead resolves the target (default: latest stable `minds-v*` tag; override honored), checks single-flight + a clean tree, and **dispatches a background worker** — it never merges in the served tree itself.
- If another update-self pass is already in flight, or the tree is dirty, the lead **surfaces and stops** instead of proceeding.
- The worker fetches the target, runs a trial merge to enumerate real conflicts, and:
  - **Auto-resolves clear-cut conflicts** — agent-specific files (`PURPOSE.md`, agent-specific `CLAUDE.md` sections) keep local; files untouched locally take upstream.
  - **Surfaces only genuine, unresolvable incompatibilities** via a `question` gate (e.g. local and upstream both rewrote the same `system_interface` backend in conflicting ways). The lead relays them to the user; the worker resumes with the resolution. Ideally this happens from diff analysis before committing to the merge.
- The worker completes the merge on its own branch, preserving the `update-self:` merge-commit subject.
- The worker classifies the **merged set** (files where local ends up different from upstream) vs **pulled-in set** (clean upstream arrivals). **Validation depth** is scoped to the merged set: full suite + `npm run lint` / `npm run test` + ratchets for each impacted project, isolated-service boots for impacted services, and Playwright live-testing of **any impacted web surface — `system_interface` or a user web service — when its merge was non-trivial** (a real integration between local and upstream changes, not a clean pull). The isolated boot uses `serve_isolated_instance.py` against a scratch data copy so the live store is never written. A clean pull-in is trusted (already tested upstream) and not re-validated — **but is still applied**: if it touches a live runtime dependency (a shared script/tool a service uses), the dependent services are still restarted at reveal so the new code takes effect.
- How deep the downstream-consumer trace goes is **worker judgment** — direct `supervisord.conf` `command` references are deterministic; transitive or scheduled-invocation dependencies are reasoned about, with a bias toward restarting a service when in doubt rather than leaving it on stale code.
- The worker **generates a "what's new" report** (new `changelog/` entries in the target range + its own notes on conflicts, resolutions, impacted services, and any rebuild-only changes) and reports `done`.
- The lead presents that summary and the merge resolutions to the user as a **hard approval gate** — nothing touches the live workspace until the user approves.
- On approval, the lead **reveals by change class**:
  - `system_interface` real-merge → preview tab, then `reveal_system_interface.py reveal` (rebuilds `static/`, refreshes deps on manifest change, pre-flights, health-checks, auto-rolls-back). Clean `system_interface` pull → reveal only, **preview skipped**.
  - service / `supervisord.conf` / `bootstrap` change → `mngr start --restart system-services` (bounces tmux → bootstrap → supervisord) + tab refresh; preview at lead's judgment.
  - editable-tool manifest change (`vendor/mngr`, `apps/system_interface` `pyproject.toml`) → env refresh (`uv sync --all-packages` / `uv tool install -e … --reinstall`); code-only editable changes need nothing.
  - `Dockerfile` change → the worker reasons about *what* changed. A hunk with a live-applicable equivalent is applied in the running container (e.g. a `CLAUDE_CODE_VERSION` bump → re-run `scripts/setup_system.sh`'s install step to the new pin, keeping `agent_types.claude.version` in `.mngr/settings.toml` in sync). A genuinely image-level hunk (base `FROM`, `apt-get`-baked packages, build-time layout) is merged but **flagged to the user as needing a manual workspace rebuild** to take effect. A single Dockerfile diff can be partly each.
  - skills / `CLAUDE.md` / `.agents/shared/` / `scripts/` → merged; new versions apply to future agents automatically **unless a live service or runner depends on the changed file at runtime** (e.g. a `supervisord` program's `scripts/*.py`, or a crystallized pipeline's script under `.agents/skills/<skill>/scripts/` invoked by a scheduled job). In that case the worker traces the downstream consumer and the change is validated and revealed like a service change (restart the affected service), rather than silently merged.
- If the worker gets stuck, can't reconcile after the user's resolution, or is OOM-shed, the lead surfaces it via the standard worker-failure path. **Nothing is merged or revealed; the live workspace is untouched.**
- After a successful reveal, the worker is destroyed and the tracking ticket closed, consistent with the other harden flows.

## Changes

  - Target resolution (default latest stable `minds-v*`, override to tag/`main`), single-flight check, clean-tree precondition.
  - Dispatch a background worker via `launch-task` machinery with an update-self task file (own branch + worktree, `lead_agent` / `finish_report_path`, worker-reporting protocol).
  - Lead-proxy handling of the `question` gate (relay conflicts to user, pass resolution back) and terminal statuses (`done` → approval gate → reveal; `stuck`/dead → worker-failure surfacing, nothing revealed).
  - Post-approval reveal dispatch by change class (delegating `system_interface` to `reveal_system_interface.py`, services to `mngr start --restart system-services` + refresh, editable-tool env refresh on manifest change, Dockerfile → surface-for-rebuild).
  - Preserve the `update-self:` merge-commit subject convention; document keep-local / take-upstream auto-resolution rules and the "surface only genuine ambiguities" bar.

- **Generalize existing shared machinery as needed** so this flow reuses rather than forks it:
  - `serve_isolated_instance.py` — boot any impacted service in isolation for the worker's boot checks (already parameterized; confirm it covers the services this flow touches).
  - `reveal_system_interface.py` — reused as-is for the `system_interface` reveal; confirm the preview-skip path (clean pull → reveal without preview) is expressible via the lead's orchestration without script changes.
  - `create_worker.py` (`launch` / `await`) and lead-proxy / worker-reporting references — reused for dispatch, gating, and report polling.

- **Docs:** refresh the `update-self` skill description and any README/CLAUDE.md pointer that describes the old blind-pull behavior, so the canonical path is the new safe flow.

## Provisioning changes (`scripts/setup_system.sh`, `.mngr/**`) — current handling and follow-up

- **The gap this closes.** Provisioning files — the pinned-toolchain scripts (`setup_system.sh`, `install_secret_scanners.sh`, `_provision_guard.sh`) and the `mngr create` config under `.mngr/**` — take effect at image-build / workspace-create / first-boot time, so a change to one never reaches a *live* workspace by restarting a service (nothing running imports it). Left in `shared_runtime` / `other`, the impact analysis found no live consumer and concluded "nothing to reveal," so e.g. a `LATCHKEY_VERSION` bump or a settings change was merged but silently *not applied* until the next full rebuild. These now form a dedicated `provisioner` reveal class (`classify_path` in `update_self.py`, unit-tested).
- **Current best-effort application (shipped).** The worker examines each provisioning change and produces a per-change apply plan the lead executes:
  - Toolchain-pin bumps (`LATCHKEY_VERSION`, …) → live-applicable by re-running the idempotent `setup_system.sh` (the merge changes the repo tree, so the content-addressed provision guard no longer skips).
  - `.mngr/settings.toml` changes → the merged file governs all *future* creates automatically (new workspaces + `launch-task` sub-agents), but the *current* workspace was built under the old settings, so the worker **best-effort mirrors each change into a live counterpart** within the merge and the lead runs the named restart (`mngr start --restart system-services`, provisioner re-run). This is deliberately aggressive: env vars and agent behavior (`host_env` / `pass_env` / `settings_overrides` / `disable_plugin`) are fiddly but live-applyable (mirror into a `profile.d` entry / supervisord `environment=` / what the agent reads, then bounce the consumers); toolchain/version pins mirror into `setup_system.sh` / the Dockerfile; `extra_provision_command` is run live. Lockstep pins kept consistent.
  - Only a **container build/launch parameter a running container can't adopt** — a `[create_templates.*]` / `[providers.*]` `build_arg`, `start_arg` (`--security-opt`, `--tmpfs`, `--cpus`, …), or runtime flag (`runsc` / `docker_runtime`) — is genuinely rebuild-only for the current workspace → flagged to the user, like an image-level `Dockerfile` hunk.
  - **Global-dependency bump with a dependent (safety turns on provenance).** When a merge bumps a *global* dependency (a `setup_system.sh` / `install_secret_scanners.sh` / Dockerfile toolchain pin), whether it is safe to hot-apply depends on **who consumes the new version**, because the worker's worktree can't itself validate the pair (worktree isolation isolates the repo tree, not the host-global toolchain; its env still has the old dep, and isolated service boots share that same toolchain). Split: **built-in dependent** (`system_interface`, a `libs/*` service — code the same upstream release tested against the new dep) → **apply live**, on the same trust-upstream basis as any pulled-in change; **user-created dependent** (a local service / web app / script the workspace built) → upstream never tested it and the worker can't either → **rebuild-only** (or `stuck` if leaving it unapplied would break the running workspace), because a workspace recreate is the only way to provision the new substrate and re-run the user code against it together. In all cases the worker **researches the version delta online** (real release notes, not memory) and **reports the coupling + any validation gap explicitly** rather than implying coverage.
  - **Backup first.** The flow's first precondition takes an on-demand full-workspace backup (`uv run host-backup-now`) so the pass — whose reveal re-runs provisioners and restarts services — is recoverable; if backups aren't configured, the lead surfaces that and gets explicit go-ahead before proceeding without a restore point.
  - **Escape hatch:** a change the worker can neither apply live nor safely defer to a rebuild without breaking the running workspace → the worker reports `stuck` and refuses the update, leaving the live workspace untouched.
- **Follow-up (not done here):** a dedicated **settings/create-template migration process** that applies `.mngr/**` changes to an already-provisioned workspace cleanly — deriving the live-apply actions from a declarative mapping (or an in-place workspace re-provision) rather than the worker's per-change best-effort mirroring — so create-template changes stop being rebuild-only and the best-effort/`stuck` heuristics can be retired. **This is also where the coupled-dependency robustness gap gets closed properly:** faithful validation of a coupled global-dep + dependent-service change needs *environment* isolation (a fresh, fully-provisioned scratch container/VM built from the worker's branch — `mngr create`-style — not just an isolated service boot on the shared host toolchain), and safe application needs a **staged recreate** (or blue/green) rather than an in-place substrate mutation. Until then, coupled substrate changes are deliberately routed to rebuild-only / `stuck` rather than hot-applied.
