# Agent-driven workspace layout operations

## Overview

- Broaden the existing agent-facing tab helper (`scripts/web_view.py` -- today: `list/open/refresh`) into a general layout-mutation surface so agents can surface any view, close panels, focus, split, move, rename, maximize/restore, and replace iframe URLs. Rename to `scripts/layout.py` to match the broader scope.
- Every op except `replace-url` is state-preserving: no sibling iframe reloads, no full-window refresh. The motivation is that user state (in-progress chat input, web-app form state) lives in iframes that today have no draft-save story; a layout change must not destroy that state. A future shared-draft-state primitive will eventually make this concern moot, but until then structured ops are the safe path -- agents must not edit `layout.json` and trigger a full reload.
- Add a separate `inspect` op that describes the live layout (group structure, ordering, spatial relations, coarse geometry, active state, per-panel `{ref, title, panel_type, service_name?}`). Anchor-style args like `--relative-to <ref>` are meaningless without it. Output is YAML by default (best LLM-accuracy/token tradeoff per current research on agent-state serialization), with `--json` for programmatic consumers.
- Coordinate concurrent agent requests via an advisory in-process mutex on the workspace_server: a 500 ms window per layout-mutating broadcast. Conflicting requests fail with an HTTP 409 that names the in-flight op and its originating agent so the rejected caller can decide whether to retry. Read-only ops (`inspect`, `list`, `refresh`, `focus`) bypass the lock. (Wait -- `focus` mutates serialized layout, so it does acquire the lock; see Changes.)
- Wire the new ops into a new `manage-layout` skill that any agent can invoke; `build-web-service` keeps recommending `open` for the simple visible-UI case and defers to the full skill only when an agent needs more.

## Expected behavior

- An agent runs `layout.py inspect` and gets a compact YAML rendering of the current dockview state: each group's panels in tab order, which panel is active, coarse split orientation between groups (left/right/above/below adjacency), and rough size ratios. Every panel carries a stable `ref` like `service:web` / `chat:flying-otter` / `terminal:<short-hash>` / `url:<short-hash>` / `subagent:<session-id>`.
- An agent runs `layout.py list` and gets a flat YAML enumeration of every addressable thing in this workspace -- registered services + mngr-level agents -- with `{ref, kind, display_name, is_open, is_running}` per entry. `is_open` reflects current panel presence; `is_running` reflects whether the underlying service/agent process is live.
- `layout.py open <name>` continues to work as before (focus-if-already-open, else split right of caller's chat at 60/40), but is now implemented as the composition of `focus` and `split` so improvements to those primitives flow through to `open` automatically. Bare service-name shorthand still resolves: `layout.py open web` and `layout.py open service:web` behave identically.
- `layout.py split <new-service> [--relative-to <ref>] [--direction left|right|above|below] [--ratio N]` opens a new iframe panel as a split. Defaults: `--relative-to=<caller's chat panel>`, `--direction=right`, `--ratio=0.6`. Existing sibling panels are NOT reloaded; only the new panel mounts.
- `layout.py focus <ref>` activates the named panel within its group. No-op error if the ref doesn't exist (HTTP 404, plain `{detail}` -- the agent calls `inspect` to recover).
- `layout.py close <ref>` removes the panel. Any panel is closable, including the primary mngr agent's chat -- dockview already auto-reopens it if the user is left with zero panels.
- `layout.py move <ref> --relative-to <other> --direction ...` relocates an existing panel via dockview's `moveGroup`/`movePanel`. The iframe DOM node is preserved, so no reload.
- `layout.py rename <ref> <new-title>` updates the tab title. Persists through `layout.json` autosave.
- `layout.py maximize <ref>` and `layout.py restore` call dockview's `setMaximizedGroup` / `exitMaximizedGroup`; underlying layout content is preserved during the maximize.
- `layout.py replace-url <ref> <url>` swaps the iframe `src`. Only the targeted iframe reloads; sibling panels untouched. Accepted URLs: `service:<name>[/<path>]` shorthand (expands to `/service/<name>/<path>`) OR a full `https://...` external URL. Anything else is rejected with a clear error.
- `layout.py refresh <ref>` reloads the iframe at the named panel without touching siblings. The legacy `refresh <service-name>` shorthand still works as `refresh service:<name>` and reloads every iframe tied to that service across the layout.
- Concurrent agents: if agent A invokes `split` while agent B's `move` is already in flight (within the 500 ms window), agent A's request fails with HTTP 409. The error response includes `{detail, retry_after_ms, in_flight: {agent_id, operation, args, started_at}}` so A can log who's mid-operation and pick a retry strategy. The script does not auto-retry -- it surfaces the 409 to the agent.
- Caller's `MNGR_AGENT_ID` is sent as a request header and recorded in workspace_server logs alongside the op. v1 has no enforcement based on this tag; it is telemetry/debug surface for future restrictions.
- For harness-level subagents (Claude Code Task sub-sessions), `MNGR_AGENT_ID` is inherited from the parent mngr agent, so `--relative-to=self` resolves to the parent mngr agent's chat panel. This matches the dockview model where subagents don't own a dedicated chat tab; they render inside their parent's chat / subagent-view panel.
- The `build-web-service` skill prompt continues to recommend `layout.py open <name>` after verification for visible-UI services. When an agent wants something beyond "open this service," the skill points to the new `manage-layout` skill for the full op set.

## Changes

- Rename `scripts/web_view.py` -> `scripts/layout.py`. Replace the three-subcommand surface with: `list`, `inspect`, `open`, `focus`, `split`, `close`, `move`, `rename`, `maximize`, `restore`, `replace-url`, `refresh`. Keep argument-parsing minimal; reuse the existing loopback-POST + `_OPEN_REGISTRATION_TIMEOUT_SECONDS` polling pattern for ops that target services. Add a header `X-Mngr-Agent-Id` populated from `MNGR_AGENT_ID` on every POST.
- Add unified ref resolution: a `service:` / `chat:` / `terminal:` / `url:` / `subagent:` prefix scheme. Stable refs for ad-hoc panels (terminal/url) derive from a short hash of the dockview panel id so they don't renumber when other panels close. Accept bare service-name shorthand (`web` -> `service:web`) on every op that takes a ref for ergonomic continuity with today's `open`/`refresh`.
- Add the `inspect` source path: workspace_server reads its own persisted `layout.json` at `$MNGR_HOST_DIR/agents/$MNGR_AGENT_ID/workspace_layout/layout.json` and serializes a compact summary (group tree, panel-order-within-group, active panel, coarse adjacency between groups, ratio buckets). The frontend's `DockviewWorkspace.ts` already auto-saves this file with a 1.5 s debounce; accept that staleness as the worst case. Format the output as YAML by default with stable refs, `--json` flag for raw object.
- Replace the loopback endpoints `/api/refresh-service/<name>/broadcast` and `/api/open-tab/<name>/broadcast` with a single richer endpoint: `POST /api/layout/broadcast` carrying `{op, args, agent_id}`. The workspace_server validates loopback, validates op-name, acquires the advisory mutex if the op is mutating, then dispatches to a per-op broadcaster method. Remove the two older endpoints atomically in this PR -- no deprecation cycle, since the only known caller is this same script and the WS contract is internal to the container.
- Add an in-process advisory mutex to `apps/system_interface/imbue/minds_workspace_server/server.py` (or a small `layout_mutex.py` next to it). The mutex tracks `{holder_agent_id, op, args, started_at}` and auto-releases after a 500 ms TTL. Mutating ops attempt to acquire; if held, the endpoint returns HTTP 409 with the holder metadata. Non-mutating ops (`inspect`, `list`, `refresh`) bypass entirely. `focus` ACQUIRES because it mutates serialized active-panel state.
- Restructure `WebSocketBroadcaster` (`apps/system_interface/imbue/minds_workspace_server/ws_broadcaster.py`) around a single `broadcast_layout_op(op, args)` method that emits `{type: "layout_op", op, args}` to all connected frontends. The existing `broadcast_open_tab` and `broadcast_refresh_service` methods are folded into this single entry point (callers use the unified method with `op="open"` / `op="refresh"` instead).
- Extend the frontend layer:
    - `apps/system_interface/frontend/src/models/AgentManager.ts`: replace today's `addOpenTabListener` / `addRefreshServiceListener` with a single `addLayoutOpListener` that dispatches per-op typed callbacks. Delete the two narrower listener registrations -- there are no other consumers (the WS message names change atomically with this PR).
    - `apps/system_interface/frontend/src/views/DockviewWorkspace.ts`: add handlers for each layout op. Use dockview's existing primitives -- `addPanel` with `position: { referencePanel, direction }` for `split`, `movePanel` / `moveGroup` for `move`, `setActivePanel` for `focus`, `removePanel` for `close`, `setMaximizedGroup` / `exitMaximizedGroup` for `maximize`/`restore`, panel title API for `rename`, and `IframePanel`-level URL prop for `replace-url`.
    - Add a ref-resolution helper in the frontend that converts `service:` / `chat:` / `terminal:` / `url:` / `subagent:` refs into the corresponding dockview panel id by inspecting `panelParams`. Centralizing here avoids drift between the inspect output and the mutation handlers.
- Add a `manage-layout` skill at `.agents/skills/manage-layout/SKILL.md` documenting the full op set with examples and the ref scheme. Update `.agents/skills/build-web-service/SKILL.md` to keep recommending `layout.py open <name>` for the simple post-verification case and point to `manage-layout` for everything else.
- Error responses: HTTP 404 with `{detail}` only on unknown refs; HTTP 409 with `{detail, retry_after_ms, in_flight}` on mutex conflict; HTTP 400 with `{detail}` on bad URL / bad direction / bad op-name. The script translates each to a distinct non-zero exit code and a stderr message that's actionable for an agent.
- Tests:
    - Backend unit/integration tests covering each new endpoint, lock acquire/release semantics, 409 payload shape, the unified broadcast endpoint's per-op routing, and ref-prefix validation.
    - Broadcaster tests for each new `broadcast_*` method.
    - Frontend unit tests for each layout-op handler in `DockviewWorkspace.ts`, mocking dockview, including the ref-resolution helper.
    - Script unit tests for argparse, ref-shorthand expansion, URL-shorthand validation in `replace-url`, and the loopback POST + header-emission path (mock HTTP server).
    - Add an acceptance test (`apps/system_interface/.../test_layout_pipeline.py`, marked `@pytest.mark.acceptance`) exercising the end-to-end agent -> workspace_server -> frontend pipeline for at least `inspect` + `open` + `close`. Use the workspace_server fixtures and a headless browser harness if one already exists in this repo; otherwise scope this test to backend-only verification (script -> endpoint -> broadcaster) and leave the WS-to-DOM step to manual verification.
- The desktop-client refresh teardown from the prior PR stays as-is. The frontend's old `refresh_service` and `open_tab` WS handlers are deleted and replaced by the unified `layout_op` dispatcher.
- Future-but-not-this-PR: the shared primitive input components that auto-save/restore draft state. When those land, the state-preservation argument for structured ops weakens, but the multi-agent coordination argument remains -- so the structured op surface still pays for itself.
