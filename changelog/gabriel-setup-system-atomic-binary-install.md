Fixed `scripts/setup_system.sh` failing when re-run on a live workspace. It installed the `ttyd` and `cloudflared` binaries with `curl -fsSL ... -o /usr/local/bin/<name>`, an in-place overwrite that truncates the target. When the target is a currently-running executable -- the `terminal` service runs `ttyd`, and `cloudflared` runs whenever a tunnel is active -- Linux refuses the truncating write with `ETXTBSY` (curl exit 23), and because the script is `set -e` it aborted before the later steps (latchkey, modal, secret scanners) ran. The `update-self` reveal flow re-runs the provisioner on a live host, so it hit this every time.

`setup_system.sh` now installs downloaded binaries through a new `install_downloaded_binary` helper that fetches to a temp file beside the target and `mv`s it into place. `rename(2)` over a busy executable is allowed -- running processes keep the old inode while new execs pick up the replacement -- so the provisioner is safe to re-run on a live host. The temp file shares the target's directory so the `mv` is a same-filesystem atomic rename, and the explicit `chmod 0755` reproduces the old `curl -o` (0644) + `chmod +x` result.
