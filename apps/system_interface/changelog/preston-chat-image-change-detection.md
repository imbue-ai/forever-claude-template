Fixed stale caching of files referenced in chat markdown. Images were served straight from their on-disk path with a one-year immutable cache policy, so overwriting a referenced file made new messages show the browser's stale cached copy, and re-rendering old messages could silently show the file's new content instead of what was originally posted. Download links (non-image files) revalidated, but clicking a link in an already-posted message still handed over whatever was on disk rather than flagging that it had changed.

Chat files are now served with caching disabled through a per-message change-detecting route: the frontend rewrites inline image `src` and download-link `href` URLs to `/api/chat-files/<event_id>/<path>`, and the backend records the file's mtime and size the first time each (event, path) pair is seen -- eagerly when the session watcher emits a new message, and lazily on first fetch as a fallback. A new message always fetches the file's current content; once the file behind an already-posted message is overwritten or deleted, the backend returns a non-file response and the frontend replaces the image (or download link) with a plain notice reading "This file has been changed. Please revert your workspace or ask your agent to recover it." The notice is deliberately not the file, so it cannot be clicked to enlarge, open, or download -- it only tells the user the file is stale.
