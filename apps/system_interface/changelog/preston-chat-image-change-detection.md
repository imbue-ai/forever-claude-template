Fixed stale image caching in the chat. Images referenced by chat markdown were served straight from their on-disk path with a one-year immutable cache policy, so overwriting a previously referenced file made new messages show the browser's stale cached copy, and re-rendering old messages could silently show the file's new content instead of what was originally posted.

Chat images are now served with caching disabled through a per-message change-detecting route: the frontend rewrites inline chat image URLs to `/api/chat-images/<event_id>/<path>`, and the backend records the file's mtime and size the first time each (event, path) pair is seen -- eagerly when the session watcher emits a new message, and lazily on first fetch as a fallback. A new message always fetches the file's current content; once the file behind an already-posted message is overwritten or deleted, the backend returns a non-image response and the frontend replaces the image with a plain notice reading "This file has been changed. Please revert your workspace or ask your agent to recover it." The notice is deliberately not an image, so it cannot be clicked to enlarge or downloaded -- it only tells the user the file has changed.
