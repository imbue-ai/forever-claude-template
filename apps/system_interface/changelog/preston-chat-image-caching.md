Fixed stale caching of files referenced in chat markdown. Images referenced by an absolute on-disk path were served with a one-year immutable cache policy, so overwriting a referenced file made new messages show the browser's stale cached copy, and re-rendering old messages from a cold cache showed the file's new content instead of what was originally posted. Download links (non-image files) were not cached immutably, but clicking a link in an already-posted message still handed over whatever was on disk at click time rather than the version from when the message was posted.

Each message's referenced files are now frozen per message: the frontend rewrites inline image `src` and download-link `href` URLs to `/api/chat-files/<event_id>/<path>`, and the backend snapshots the source file's bytes (content-hashed, deduplicated copies under the mngr host dir) the first time each (event, path) pair is seen -- eagerly when the session watcher emits a new message, and lazily on first fetch. Images are served inline and other files as downloads under their original filename. A new message always fetches the file's current content, and an already-posted message keeps showing (or downloading) exactly the bytes that existed when it was posted, even if the file on disk later changes.
