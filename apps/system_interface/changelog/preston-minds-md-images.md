Agents can now show files inline in chat. An agent writes a file and references its absolute on-disk path in markdown; the system interface serves the file at that path, with no upload step. An image (`![alt](/mngr/code/runtime/chat-images/chart.png)`) renders inline in the reply; any other file linked normally (`[report](/mngr/code/runtime/chat-files/report.pdf)`) is offered as a download.

The handler hangs off the single-page-app catch-all: an existing image file in a browser-renderable format (`.png`, `.jpg`/`.jpeg`, `.gif`, `.webp`, `.avif`, `.bmp`, `.ico`, `.svg`) is streamed inline; any other existing file is streamed as an attachment (download). A path carrying an image extension with no file behind it returns a 404 (broken image) so a typo'd image never silently renders the app shell, and a path that matches no file at all falls through to the SPA so client-side routing is unaffected.

Images are served inline with an immutable, one-year cache (agents are instructed to use unique filenames per image), and SVGs are additionally served with a restrictive `Content-Security-Policy` and `X-Content-Type-Options: nosniff` so a directly-opened SVG cannot execute scripts. Non-image files are served with `Content-Disposition: attachment` plus `nosniff`, so the browser downloads them rather than interpreting them in the chat's own origin.

Clicking an inline chat image now opens it full-screen in a lightbox over a dimmed backdrop. A header shows the image's title (its alt text, or the filename) centered, with Download and close (X) icon buttons at the top right; the viewer closes on a backdrop click, the X, or Escape. Inline images are also constrained to the message column width so a large image no longer overflows the chat.
