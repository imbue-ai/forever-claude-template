- Messaging an agent (`POST /api/agents/{id}/message`) no longer runs a fresh mngr discovery on every send. `AgentManager` now retains each agent's location (host/provider) from the `mngr observe` discovery stream it already consumes -- rebuilt on every full-snapshot event and dropped on destroy events -- and exposes `get_agent_matches_by_id`. `agent_discovery.send_message` consults that live, event-fed state first and messages the agent directly via mngr's pre-resolved `agents_to_message` API; it falls back to a full discovery only when the location is unknown (e.g. a just-created agent before the next snapshot) or when the known location no longer reaches the agent. The result is fresh on both agent creation and removal (the observe stream updates it), with no staleness window for the common case, and discovery stays as a correctness backstop.
- The message endpoint now addresses the agent **by id end-to-end** instead of resolving the id to a name and matching by name. `send_message` accepts an `AgentNameOrId`; the endpoint (which already has the id) passes the id, so the location lookup is an O(1) id hit and the message targets the exact agent rather than every agent sharing its name. The welcome-resend path, which only knows a name, still passes a name.
